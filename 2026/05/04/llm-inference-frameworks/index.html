大模型推理框架对比：vLLM、TensorRT-LLM 与 llama.cpp

最后更新于2026年6月12日 下午

训练好的大模型如何高效部署？本文对比三个主流推理框架的原理与适用场景。

1. 推理的核心挑战

大模型推理面临两个主要瓶颈：

内存瓶颈： KV Cache 随序列长度线性增长

计算瓶颈： 自回归解码每次只生成一个 token，GPU 利用率极低（通常 < 30%）

2. vLLM：PagedAttention 革命

2.1 核心创新：PagedAttention

效果： 内存利用率从 20-40% 提升到 90%+，吞吐量提升 2-4 倍。

新请求可以在任意时间步插入，GPU 始终满负荷运行。

2.3 适用场景

适合： 在线服务、高并发 API、需要最大化吞吐量的场景

3. TensorRT-LLM：NVIDIA 的极致优化

TensorRT-LLM 是 NVIDIA 专为其 GPU 优化的推理框架，通过算子融合、量化等手段压榨硬件极限。

3.1 核心优化技术

算子融合（Kernel Fusion）：

In-flight Batching： 类似 vLLM 的连续批处理，但针对 NVIDIA GPU 深度优化。

量化支持：

量化方式 精度损失 内存节省 速度提升
FP16 基准 基准 基准
INT8 极小 50% 1.5-2x
INT4 小 75% 2-3x
FP8 极小 50% 1.5-2x

适合： NVIDIA GPU 生产环境，追求极致延迟的场景

4. llama.cpp：CPU 上的大模型

llama.cpp 用纯 C++ 实现，支持在 CPU（甚至手机）上运行大模型。

4.1 核心技术：GGUF 量化格式

Q4_K_M 量化原理： 将 FP16 权重分组，每组用 4bit 整数 + 缩放因子表示，精度损失极小但内存减少 75%。

5. 三框架横向对比

维度 vLLM TensorRT-LLM llama.cpp
硬件要求 NVIDIA GPU NVIDIA GPU CPU/GPU 均可
吞吐量 极高 最高 低
延迟 低 最低 高
部署难度 简单 复杂 极简单
量化支持 AWQ/GPTQ INT4/INT8/FP8 GGUF Q2-Q8
适用场景 在线服务 生产极致优化 本地/边缘

总结

  • 云端高并发服务 → vLLM（易用 + 高吞吐）
  • NVIDIA GPU 生产环境追求极致 → TensorRT-LLM
  • 本地运行 / 无 GPU → llama.cpp
